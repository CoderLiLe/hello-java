# 分库分表深入实战

## 1. 分库分表概述

### 1.1 为什么需要分库分表
- 单表数据量过大（> 1000 万）
- 数据库写入瓶颈
- 单库连接数耗尽
- 索引深度过大

### 1.2 拆分方式
| 方式 | 说明 | 适用 |
|------|------|------|
| 垂直分库 | 按业务拆分到不同库 | 业务解耦 |
| 垂直分表 | 大表拆小表（字段拆分） | 大字段多 |
| 水平分库 | 按分片键分布到不同库 | 数据量大 |
| 水平分表 | 按分片键分布到不同表 | 单表大 |

## 2. 分片策略

### 2.1 常见策略
```java
// 取模分片
int dbIdx = userId % DB_COUNT;
int tableIdx = userId % TABLE_COUNT;

// 范围分片
if (userId < 1000000) -> db0;
else if (userId < 2000000) -> db1;

// 时间分片
String tableName = "order_" + LocalDate.now().format(DateTimeFormatter.ofPattern("yyyyMM"));

// 一致性哈希
TreeMap<Long, String> nodeMap = new TreeMap<>();
// 添加虚拟节点
for (int i = 0; i < 160; i++) {
    nodeMap.put(hash("node-" + i), node);
}
```

### 2.2 分片键选择
- 高频查询条件（WHERE =）
- 数据分布均匀
- 不轻易变更

## 3. 分布式事务

### 3.1 方案对比
| 方案 | 一致性 | 性能 | 复杂度 |
|------|--------|------|--------|
| 2PC | 强 | 低 | 高 |
| TCC | 强 | 中 | 高 |
| 补偿 | 最终 | 高 | 中 |
| MQ | 最终 | 高 | 低 |

### 3.2 可靠消息最终一致性
```java
// 1. 本地事务 + 消息表
@Transactional
public void createOrder(Order order) {
    orderMapper.insert(order);           // 本地事务
    messageMapper.insert(msg);           // 同一事务
}

// 2. 定时任务扫描发送
@Scheduled(fixedDelay = 5000)
public void sendMessage() {
    List<Message> msgs = messageMapper.selectPending();
    for (Message msg : msgs) {
        if (mqSender.send(msg.getContent())) {
            messageMapper.updateStatus(msg.getId(), "SENT");
        }
    }
}
```

### 4.1 雪花算法
```java
public class SnowflakeIdWorker {
    private long workerId;
    private long sequence = 0L;
    private long lastTimestamp = -1L;
    
    public synchronized long nextId() {
        long timestamp = System.currentTimeMillis();
        if (timestamp < lastTimestamp) {
            // 时钟回拨处理
        }
        if (lastTimestamp == timestamp) {
            sequence = (sequence + 1) & (-1L >>> 12);
            if (sequence == 0) {
                timestamp = waitNextMillis(lastTimestamp);
            }
        } else {
            sequence = 0L;
        }
        lastTimestamp = timestamp;
        return ((timestamp - 1288834974657L) << 22)
                | (workerId << 12)
                | sequence;
    }
}
```

## 5. ShardingSphere

### 5.1 核心功能
- **分库分表**：水平/垂直拆分
- **读写分离**：主从自动路由
- **数据加密**：透明加解密
- **影子库**：压测数据隔离
- **分布式事务**：XA/SEATA
- **弹性迁移**：数据伸缩

### 5.2 配置
```yaml
spring:
  shardingsphere:
    datasource:
      names: ds0, ds1
      ds0:
        type: com.zaxxer.hikari.HikariDataSource
        url: jdbc:mysql://localhost:3306/ds0
      ds1:
        url: jdbc:mysql://localhost:3306/ds1
    
    rules:
      sharding:
        tables:
          t_order:
            actualDataNodes: ds${0..1}.t_order_${0..1}
            tableStrategy:
              standard:
                shardingColumn: order_id
                shardingAlgorithmName: table-inline
            databaseStrategy:
              standard:
                shardingColumn: user_id
                shardingAlgorithmName: db-inline
        shardingAlgorithms:
          table-inline:
            type: INLINE
            props:
              algorithm-expression: t_order_${order_id % 2}
          db-inline:
            type: INLINE
            props:
              algorithm-expression: ds${user_id % 2}
```

### 5.3 分布式查询
```sql
-- 支持跨库聚合查询
SELECT user_id, COUNT(*) 
FROM t_order 
WHERE status = 'PAID'
GROUP BY user_id;

-- 支持分页排序
SELECT * FROM t_order 
ORDER BY create_time DESC 
LIMIT 10 OFFSET 20;
```

## 6. 数据迁移

### 6.1 平滑迁移方案

```
   |            |              |

```

### 6.2 迁移工具
```bash

# binlog 增量同步（canal）
canal:
  instance: example
  filter: db.table
  destination: kafka/rocketmq
```

## 7. 最佳实践

- 分片键放在 WHERE 最左
- 避免跨库 JOIN
- 分布式 ID 全局唯一
- 容量规划要留余量
- 监控慢查询和分布式查询
- 分库数建议 2 的幂次方
- 做好降级和限流
