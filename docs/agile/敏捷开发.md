# 敏捷开发

## 1. 敏捷概述

敏捷开发是一种迭代式软件开发方法论，强调人员协作、快速响应变化和持续交付价值。

### 1.1 敏捷宣言
- **个体和互动** 高于 流程和工具
- **可工作的软件** 高于 详尽的文档
- **客户合作** 高于 合同谈判
- **响应变化** 高于 遵循计划

### 1.2 十二原则
1. 最高优先级：尽早、持续交付有价值软件
2. 欢迎需求变化
3. 频繁交付可工作软件（几周~几个月）
4. 业务与开发每天合作
5. 激励个人，给予信任
6. 面对面沟通最高效
7. 可工作软件是进度的主要度量
8. 可持续的开发节奏
9. 持续关注技术卓越和良好设计
10. 简化——最大化未做之事
11. 自组织团队
12. 定期反思调整

## 2. Scrum 框架

### 2.1 角色
- **Product Owner**：负责产品待办列表，最大化产品价值
- **Scrum Master**：确保 Scrum 被正确理解和使用
- **开发团队**：自组织，跨职能，3-9 人

### 2.2 工件
- **Product Backlog**：产品待办列表
- **Sprint Backlog**：当前 Sprint 任务
- **Increment**：可交付的产品增量

### 2.3 事件（活动）
- **Sprint**：1-4 周迭代
- **Sprint Planning**：Sprint 计划会
- **Daily Scrum**：每日站会（15 分钟）
- **Sprint Review**：Sprint 评审会
- **Sprint Retrospective**：Sprint 回顾会

### 2.4 Sprint 工作流
```
Product Backlog -> Sprint Planning -> Sprint Backlog
                                            |
                         Daily Scrum <-- 开发工作
                                            |
                    Sprint Review <-- Increment
                    Sprint Retrospective
```

## 3. XP（极限编程）

### 3.1 核心实践
- **结对编程**：两人一台机器
- **TDD**：测试驱动开发
- **重构**：持续改进代码
- **持续集成**：每日多次集成
- **简单设计**：只做必要设计
- **集体代码所有权**：人人可改代码
- **编码规范**：统一标准
- **隐喻**：共用系统比喻

## 4. 用户故事

### 4.1 格式
```
作为 <角色>，我想要 <功能>，以便 <价值>
```

### 4.2 三 C
- **Card**：卡片描述
- **Conversation**：对话沟通
- **Confirmation**：验收条件

### 4.3 INVEST 原则
- **I**ndependent 独立
- **N**egotiable 可协商
- **V**aluable 有价值
- **E**stimable 可估算
- **S**mall 小型
- **T**estable 可测试

## 6. 看板方法

### 6.1 核心原则
- 可视化工作流
- 限制在制品（WIP）
- 管理流动
- 明确过程策略
- 反馈循环
- 协作改进

### 6.2 看板列
```
待办 -> 分析 -> 开发 -> 测试 -> 发布 -> 完成
         WIP:3    WIP:3    WIP:2
```

## 7. 敏捷估算与度量

| 度量 | 说明 |
|------|------|
| 速率 | 每个 Sprint 完成的故事点 |
| 燃尽图 | 剩余工作量随时间变化 |
| 燃起图 | 已完成工作量 |
| 累积流图 | 各阶段工作分布 |
| 周期时间 | 从开始到完成 |
| 吞吐量 | 单位时间完成数量 |
