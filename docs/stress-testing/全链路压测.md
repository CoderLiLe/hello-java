# 全链路压测实战

## 1. 全链路压测概述

全链路压测是指在生产环境或类生产环境下，模拟真实用户请求对整个系统链路进行压力测试，验证系统在高并发下的性能和稳定性。

### 1.1 核心目标
- 验证系统容量水位
- 发现性能瓶颈
- 验证弹性伸缩能力
- 评估系统稳定性

### 1.2 与传统压测的区别
| 特性 | 传统压测 | 全链路压测 |
|------|---------|-----------|
| 环境 | 测试环境 | 生产/类生产 |
| 链路 | 单模块 | 全链路 |
| 数据 | 测试数据 | 真实/脱敏数据 |
| 影响 | 隔离 | 需防污染 |

## 2. 压测流程

### 2.1 标准流程
```
需求分析 -> 方案设计 -> 脚本开发 -> 数据准备 -> 
基线测试 -> 负载测试 -> 瓶颈分析 -> 调优验证 ->
容量评估 -> 报告输出
```

### 2.2 关键指标
| 指标 | 说明 |
|------|------|
| TPS/QPS | 每秒处理事务/请求数 |
| RT | 响应时间（avg/p99/p999） |
| 错误率 | 请求失败比例 |
| CPU/内存 | 资源使用率 |
| GC 情况 | GC 频率和时间 |
| 连接池 | 数据库/连接池使用 |

## 4. 压测方案

### 4.1 数据隔离
- **逻辑隔离**：压测数据打标（压测标识）
- **物理隔离**：独立的压测数据库
- **影子表**：同一库中影子表

### 4.2 数据标识（压测标）
```java
// ThreadLocal 传递压测标识
public class StressContext {
    private static final ThreadLocal<Boolean> STRESS_FLAG = new ThreadLocal<>();
    
    public static void setStress(boolean flag) {
        STRESS_FLAG.set(flag);
    }
    
    public static boolean isStress() {
        return Boolean.TRUE.equals(STRESS_FLAG.get());
    }
    
    public static void clear() {
        STRESS_FLAG.remove();
    }
}
```

### 4.3 流量染色
- Header：`X-Stress-Test: true`
- 数据库：影子表（`_shadow` 后缀）
- 消息队列：独立的影子 Topic

## 5. 容量规划

### 5.1 容量评估
```
系统容量 = 单机容量 x 副本数 / 冗余系数
```

### 5.2 容量模型
```
PV -> QPS 转换：
QPS = PV / 统计时间 / 并发因子

示例：日 PV 1000 万，80% 在 4 小时
QPS = 10000000 * 0.8 / (4 * 3600) ≈ 556
峰值 QPS = 556 * 3 ≈ 1667
```

## 6. 性能调优方向

### 6.1 常见瓶颈
| 层次 | 瓶颈 | 解决方案 |
|------|------|---------|
| 代码 | 算法 | 优化算法/缓存 |
| JVM | GC | 调优 GC 参数 |
| DB | SQL/索引 | 优化 SQL/加索引 |
| 网络 | 带宽 | 压缩/CDN |
| 中间件 | 线程池 | 调整参数 |

### 6.2 压测策略
```python
# 阶梯加压
[10->100->500->1000->1500->2000] TPS
# 每阶段运行 5 分钟，观察系统表现

# 峰值保持
保持峰值 TPS 运行 30 分钟

# 突发测试
突然增加 3 倍流量
```

## 7. 监控体系

### 7.1 压测期间监控
- **系统监控**：CPU、内存、磁盘、网络
- **应用监控**：JVM、线程池、连接池
- **中间件监控**：MQ 堆积、DB 连接
- **业务监控**：订单量、错误率

### 7.2 告警阈值
- CPU > 80%
- RT P99 > 1000ms
- 错误率 > 1%
- GC 频率过高

## 8. 最佳实践

- 压测前通知所有相关团队
- 准备自动化回滚方案
- 逐步加压，从低到高
- 全链路压测常态化（大促前必做）
- 建立性能基线，持续对比
- 压测数据及时清理
- 关注第三方依赖的限流情况
