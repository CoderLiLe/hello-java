# 领域驱动设计（DDD）

## 1. DDD 概述

DDD（Domain-Driven Design）是由 Eric Evans 提出的软件开发方法论，核心是聚焦业务领域，通过领域模型驱动软件设计。

### 1.1 核心原则
- **聚焦核心领域**：将资源集中在核心业务
- **模型驱动设计**：代码即模型
- **统一语言**：团队共享业务术语
- **限界上下文**：明确的模型边界

## 2. 战略设计

### 2.1 限界上下文（Bounded Context）
每个上下文有明确的职责边界和统一的领域语言。

```
[订单上下文] <--> [库存上下文] <--> [支付上下文]
     |                  |                |
[配送上下文]      [仓储上下文]     [账务上下文]
```

### 2.2 上下文映射关系
| 关系 | 说明 |
|------|------|
| 合作关系 | 两个团队紧密协作 |
| 共享内核 | 共享通用模型 |
| 客户-供应商 | 上游影响下游 |
| 防腐层 | 隔离外部模型 |
| 开放服务 | 作为服务被调用 |
| 各行其道 | 无关联 |

### 2.3 实体与值对象
```java
// 实体 - 有唯一标识
public class Order {
    private OrderId id;        // 值对象
    private String orderNo;
    private List<OrderItem> items;
    private OrderStatus status;
    
    public void changeStatus(OrderStatus newStatus) {
        // 业务规则
        if (this.status == OrderStatus.PAID && newStatus == OrderStatus.SHIPPED) {
            this.status = newStatus;
        }
    }
}

// 值对象 - 无标识，不可变
public class Address {
    private final String province;
    private final String city;
    private final String detail;
    // 构造器、equals、hashCode
}
```

### 2.4 聚合（Aggregate）
- 聚合根：外部只能通过聚合根操作内部实体
- 保证数据一致性
- 通过 ID 引用其他聚合

```java
// 聚合根
public class Order {
    private List<OrderItem> items;
    
    public void addItem(ProductId productId, int quantity, Money price) {
        // 业务校验
        OrderItem item = new OrderItem(productId, quantity, price);
        items.add(item);
    }
}
```

### 2.5 领域服务
```java
// 跨聚合、跨限界上下文的业务逻辑
public class TransferService {
    @Transactional
    public void transfer(Account from, Account to, Money amount) {
        from.debit(amount);
        to.credit(amount);
    }
}
```

## 3. 战术设计

### 3.1 分层架构
```
Interface（控制器、DTO）
    |
Application（应用服务）
    |
Domain（实体、值对象、聚合、领域服务）
    |
Infrastructure（仓储实现、消息、ORM）
```

### 3.2 仓库（Repository）
```java
public interface OrderRepository {
    Order findById(OrderId id);
    void save(Order order);
    void delete(OrderId id);
}
```

### 3.3 工厂（Factory）
```java
public class OrderFactory {
    public Order createOrder(CustomerId customerId, List<OrderItem> items) {
        Order order = new Order(new OrderId(UUID.randomUUID().toString()));
        items.forEach(order::addItem);
        return order;
    }
}
```

### 3.4 领域事件
```java
// 事件定义
public class OrderPlacedEvent implements DomainEvent {
    private final OrderId orderId;
    private final Instant occurredOn;
}

// 发布事件
public class Order {
    private List<DomainEvent> events = new ArrayList<>();
    
    public void place() {
        this.status = OrderStatus.PLACED;
        events.add(new OrderPlacedEvent(this.id, Instant.now()));
    }
}
```

## 4. CQRS

CQRS（命令查询职责分离）将读写模型分离。

```java
// 命令 - 修改数据
public class CreateOrderCommand implements Command {
    private String customerId;
    private List<OrderItem> items;
}

// 查询 - 读取数据
public interface OrderQueryRepository {
    OrderSummary findOrderSummary(OrderId id);
    List<OrderListDTO> findOrders(CustomerId customerId);
}

// 命令处理器
@Component
public class CreateOrderHandler {
    @CommandHandler
    public void handle(CreateOrderCommand cmd) {
        // 业务逻辑，发布事件
    }
}
```

## 5. 事件溯源

- 存储领域事件序列而非当前状态
- 重放事件重建聚合状态
- 天然审计日志

```java
public class Order {
    private List<Event> changes = new ArrayList<>();
    
    public static Order recreate(List<Event> events) {
        Order order = new Order();
        events.forEach(order::apply);
        return order;
    }
    
    private void apply(Event event) {
        if (event instanceof OrderPlacedEvent) {
            this.status = OrderStatus.PLACED;
        }
    }
}
```

## 6. 微服务与 DDD

| DDD 概念 | 微服务映射 |
|----------|-----------|
| 限界上下文 | 一个微服务 |
| 聚合 | 事务边界 |
| 上下文映射 | 服务间通信 |
| 通用语言 | API 契约 |
| 实体 | 业务对象 |
| 值对象 | 不可变类 |
| 领域事件 | 异步消息 |

## 7. 最佳实践

- 从事件风暴（Event Storming）开始领域建模
- 聚合设计要小，一个事务只修改一个聚合
- 跨聚合操作使用领域事件异步处理
- 仓库只对聚合根操作
- 避免在领域层使用 Spring 注解
- 分层清晰，依赖方向由外向内
